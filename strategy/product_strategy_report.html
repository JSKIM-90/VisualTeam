RENOBIT 제품 전략 보고서

대시보드 중심 플랫폼으로의 집중 전략
2025년 10월

Executive Summary

현황: RENOBIT은 범용 개발도구로 안정화되었으나, 명확한 타겟팅 없이 프로젝트별 커스터마이징에 대응하다 보니 React와 경쟁하는 구도가 되어 효율성 저하

전략: 대시보드 영역에 집중하여 패턴화와 템플릿 구축을 통해 쉽고 빠른 프로젝트 구축 달성

근거:

  • 회사 프로젝트의 70%가 대시보드
  • RENOBIT 구조가 대시보드와 자연스럽게 일치
  • 범위를 좁혀 경쟁 우위 확보

기대효과: 개발 속도 3-10배 향상, 재사용률 80% 이상, 명확한 시장 포지셔닝

1. 현황 진단

1.1 현재 문제점

핵심 문제

범용 플랫폼으로 운영되면서 발생한 비효율

문제 원인 결과
프로젝트별 커스터마이징 늪 명확한 경계 부재 매번 처음부터 개발
React와의 불리한 경쟁 범용 vs 범용 구도 개발 속도 열세
재사용 없음 패턴화 불가능 누적된 자산 활용 못함
팀 소진 반복적 커스터마이징 효율 저하, 이직 위험

1.2 근본 원인

타겟팅의 부재

도메인 특정 없음
↓
모든 요구에 대응
↓
반복 패턴 식별 불가
↓
템플릿화 불가능
↓
매 프로젝트마다 백지 출발

"타겟팅 없이는 쉬운 사용과 빠른 개발을 달성할 수 없다"

2. 전략 방향

2.1 왜 경계가 필요한가

경계 없음의 결과

경계 없음
↓
모든 프로젝트 대응
↓
범용 프레임워크화
↓
React/Vue와 직접 경쟁
↓
현실적으로 불가능

경계 설정의 효과

명확한 경계
↓
특정 영역 집중
↓
반복 패턴 식별
↓
템플릿화/자동화
↓
쉽고 빠른 구축

핵심 원칙

경계는 제약이 아니라 강점이다

  • 좁은 영역에서 압도적 우위 확보
  • 모든 것을 하려다 아무것도 못하는 것보다 낫다
  • 선택과 집중이 경쟁력

2.2 왜 대시보드인가

2.2.1 비즈니스 근거

기준 내용 결론
프로젝트 비중 회사 프로젝트의 70%가 대시보드 최대 시장
구축 경험 다수 대시보드 구축 완료 축적된 자산
디자인 자산 템플릿/컴포넌트 보유 재사용 가능
투자 대비 효과 ROI 가장 높음 전략적 선택

2.2.2 기술적 근거: 구조적 일치

대시보드의 본질

정의: 외부 데이터 수집 + 시각적 표현 + 상태 인지 + 읽기 중심

흐름:

Data (수집)
→
System (처리)
→
User (인지)
→
Decision (판단)

핵심 특징:

  1. 외부 데이터 수집: 데이터를 생성하지 않고 가져옴
  2. 시각적 표현: 차트/테이블로 표현
  3. 상태 인지: 현재 상황 파악이 목적
  4. 읽기 중심: 조회가 90% 이상

RENOBIT 구조

핵심 구성:

Dataset Manager
↓
데이터 수집 및 관리
↓
Chart/Table 컴포넌트
↓
시각적 표현
↓
Viewer (읽기 중심 결과물)

구조적 일치

대시보드 본질 ↔ RENOBIT 구조
외부 데이터 수집 ↔ Dataset Manager
시각적 표현 ↔ Chart/Table 컴포넌트
상태 인지 ↔ 실시간 데이터 갱신
읽기 중심 ↔ Viewer 기반 결과물

→ RENOBIT은 태생적으로 대시보드 구조와 일치

2.2.3 RENOBIT이 약한 영역

반대 흐름:

Action (입력)
→
User (조작)
→
System (처리)
→
Data (저장)

약한 프로젝트 유형:

  • 복잡한 폼 처리 (입력 중심)
  • CRUD 시스템 (쓰기 중심)
  • 워크플로우 관리 (프로세스 중심)
  • 트랜잭션 처리 (저장 중심)

→ 이런 프로젝트는 React가 더 적합

3. 대시보드 정의 및 경계

3.1 판단 기준

4가지 핵심 체크

  • 외부 데이터를 수집하는가?
  • 차트/테이블로 시각화하는가?
  • 현재 상태 파악이 목적인가?
  • 읽기가 쓰기보다 압도적으로 많은가? (9:1 이상)

판단:

  • 4개 Yes → 대시보드 확정
  • 3개 Yes → 비중 확인 (차트/테이블 70% 이상 → 대시보드)
  • 2개 이하 → 대시보드 아님

3.2 대시보드 vs 비대시보드

구분 대시보드 ✓ 대시보드 아님 ✗
목적 상태 조회/모니터링 데이터 생성/관리
주요 UI 차트/테이블 (70%+) 폼/입력 (50%+)
데이터 흐름 외부 → 시스템 → 사용자 사용자 → 시스템 → 저장
사용 패턴 읽기 90% 이상 쓰기 30% 이상
예시 설비 모니터링, KPI 대시보드 회원 관리, 주문 처리

3.3 실전 적용 사례

✓ 대시보드 (RENOBIT 적합)

  • 설비 모니터링 시스템
    • 센서 데이터 수집 → 차트 표시
    • 실시간 상태 모니터링
    • 알람 발생 시 테이블 표시
  • 생산 현황 대시보드
  • 품질 분석 리포트
  • 영업 실적 대시보드
  • IoT 센서 시각화

✗ 비대시보드 (React 권장)

  • 정비 관리 시스템
    • 정비 계획 등록/수정/삭제
    • 작업자 배정 워크플로우
    • 부품 재고 관리 (CRUD)
  • 회원 관리 시스템
  • 주문 처리 시스템
  • CRM/ERP

△ 경계 케이스

관리자 패널 (Admin Panel)

  • 모니터링 70% + 설정 30%
  • 판단: 분리 검토
    • 모니터링 부분 → RENOBIT
    • 설정 부분 → 별도 페이지 (React)

4. 경계 밖 프로젝트 전략

4.1 기본 원칙

대시보드 영역을 먼저 확고히 한다

우선순위:

  1. 1순위: 대시보드 템플릿/프로세스 완성
  2. 2순위: 다른 영역 검토

4.2 솔루션 접근법

5단계 프로세스

Step 1: 문제 구체적 정의

사용자의 진짜 문제가 무엇인가?

↓
Step 2: 해결 프로그램 개발

가장 빠른 방법으로 개발 (React/Vue 등)

↓
Step 3: MVP/데모 구현

실제 작동하는 프로토타입

↓
Step 4: RENOBIT 통합 검토

3가지 기준 체크

↓
Step 5: 최종 결정

통합 / 신규 플랫폼 / React 유지

4.3 통합 판단 기준

RENOBIT 통합 3가지 기준 (AND 조건)

  • 대시보드 정의에 부합하는가?
    • 4가지 핵심 특징 충족
  • 3개 이상 프로젝트에서 재사용 가능한가?
    • 패턴화 가능성 확인
  • 기존 RENOBIT 구조 변경 없이 추가 가능한가?
    • 구조적 안정성 유지

판단:

  • 3개 모두 Yes → RENOBIT 통합
  • 1개라도 No → 신규 플랫폼 또는 React 유지

4.4 신규 플랫폼 판단 기준

신규 플랫폼 3가지 기준

  • 명확히 다른 도메인인가?
    • 대시보드와 완전히 다른 성격
  • 독립적 플랫폼으로 운영 가능한가?
    • 충분한 프로젝트 수, 지속성
  • 리소스 투입 대비 효과가 명확한가?
    • ROI 계산 가능

판단:

  • 3개 모두 Yes → 신규 플랫폼 개발 검토
  • 아니면 → 범용 도구(React/Vue) 사용

5. Pain Point 개선 전략

5.1 Pain Point 유형 분류

유형 설명 전략 부합 대응
Type 1
플랫폼 품질		 버그, 성능, 문서, 에러 메시지 ✓ 100% 무조건 개선
Type 2
학습 비용		 새 구조 학습, 초기 어려움 △ 단기 Pain
장기 이득		 개선 + 교육
Type 3
유연성 부족		 "이렇게 하고 싶은데 안 돼요" △ 경우에 따라 선택적 대응
Type 4
범위 밖 요구		 대시보드 외 기능 추가 요청 ✗ 충돌 거절 + 대안

5.2 Pain Point 필터링 프로세스

3단계 필터

SI 개발자 Pain Point 발생
↓
[1단계] 플랫폼 품질 문제인가?

Yes → 무조건 해결
No → 2단계

↓
[2단계] 대시보드 영역 내 문제인가?

Yes → 3단계
No → "경계 밖입니다. React 권장"

↓
[3단계] 패턴화 가능한가?

Yes → 해결 (템플릿에 반영)
No → "프로젝트 특수 케이스"

5.3 사례별 대응 방안

✓ 해결해야 하는 Pain Point

사례: "차트 설정이 너무 복잡해요"

  • 분석: 대시보드 핵심 기능, 패턴화 가능
  • 해결: 차트 프리셋, Quick Config, 템플릿 제공
  • 효과: 설정 시간 70% 단축

사례: "데이터 연결이 매번 헷갈려요"

  • 분석: 대시보드 핵심, 반복 패턴
  • 해결: Dataset Wizard, DB 프리셋, 명확한 에러 메시지
  • 효과: 초기 설정 시간 80% 단축

✗ 해결하지 않는 Pain Point

사례: "복잡한 폼 빌더가 필요해요"

  • 분석: 대시보드 영역 밖 (입력 중심)
  • 대응: "폼 중심 프로젝트는 RENOBIT 범위 밖입니다. React + Formik 사용 권장"
  • 이유: 경계 유지, 패턴 보호

사례: "워크플로우 엔진 추가해주세요"

  • 분석: 대시보드가 아님 (프로세스 중심)
  • 대응: "워크플로우는 별도 플랫폼 검토 필요"
  • 이유: 전략 충돌 방지

△ 조건부 대응

사례: "이 특정 프로젝트만 특이한 UI 필요"

  • 분석: 대시보드 맞음, 하지만 1회성
  • 대응: "FreeCode 컴포넌트로 직접 구현하세요"
  • 이유: 패턴화 불가능한 특수 케이스

5.4 핵심 원칙

Pain Point 개선 ≠ 모든 요구 수용

Pain Point 개선 = 대시보드 영역 내 패턴화 가능한 문제 해결

균형의 원칙:

  • 잘못된 균형: 모든 Pain Point 해결 → 범용화 → 전략 실패
  • 올바른 균형: 선택적 해결 → 패턴 강화 → 전략 성공

6. 실행 계획

6.1 4단계 로드맵

단계 기간 목표 주요 활동
Phase 1
기준 정립		 1개월 명확한 경계 설정 • 대시보드 기능 명세 확정
• 과거 프로젝트 분석
• 패턴 추출
• 템플릿 1개 개발
Phase 2
검증		 3개월 전략 유효성 확인 • 신규 프로젝트 3개 적용
• 개발 시간 측정
• 피드백 수집
• 템플릿 개선
Phase 3
확산		 6개월 전면 적용 • 템플릿 5개로 확장
• 전체 프로젝트 적용
• SI 개발자 교육
• 성과 측정
Phase 4
고도화		 1년 경쟁 우위 확보 • 자동화 구축
• Quick Start Wizard
• 플랫폼 개선
• 신규 도메인 검토

6.2 Phase 1 상세 계획 (1개월)

Week 1-2: 대시보드 기능 명세

  • 보편적 기능 정리
    • 데이터 수집 방식 (SQL, REST API, WebSocket)
    • 시각화 유형 (Chart, Table, Gauge, Map)
    • 인터랙션 패턴 (필터링, 드릴다운, 실시간 갱신)
  • 과거 프로젝트 분석
    • 지난 1-2년 대시보드 프로젝트 전수 조사
    • 공통 패턴 5-10개 추출
    • 재사용 가능 컴포넌트 목록화

Week 3-4: 첫 템플릿 개발

  • 가장 빈도 높은 패턴 선택
  • 템플릿 개발
    • 레이아웃 구조
    • 데이터 연결 구조
    • 주요 컴포넌트 조합
  • Quick Start 가이드 작성

7. 기대 효과

7.1 정량적 효과

개발 속도 향상

8. 결론

핵심 메시지

1. 경계가 필요하다

범용 플랫폼으로는 React와 경쟁할 수 없다. 명확한 경계 설정을 통해 집중하고, 좁은 영역에서 압도적 우위를 확보해야 한다.

2. 대시보드가 답이다

회사 프로젝트의 70%가 대시보드이며, RENOBIT 구조가 대시보드와 자연스럽게 일치한다. 비즈니스 근거와 기술적 적합성이 모두 확인되었다.

3. 대시보드 = 데이터 수집 + 시각화 + 읽기 중심

명확한 정의를 통해 프로젝트별 판단이 가능하다. 4가지 기준으로 빠른 의사결정을 할 수 있다.

4. 경계 밖은 솔루션 접근

모든 프로젝트를 RENOBIT으로 할 필요는 없다. 대시보드가 아니면 React를 사용하고, 필요시 통합을 검토한다.

5. 선택적 Pain Point 해결

모든 요구를 수용하면 다시 범용화된다. 대시보드 영역 내, 패턴화 가능한 문제만 해결한다.

8.1 성공 조건

  1. 명확한 경계 유지
    • 대시보드 정의를 엄격히 적용
  2. 단계적 실행
    • 급진적 변화 지양, 점진적 검증
  3. 측정 기반 개선
    • KPI 지속 모니터링, 데이터 기반 의사결정
  4. 팀 합의 확보
    • 플랫폼팀, SI팀, 경영진 모두 동의
  5. 용기 있는 거절
    • 경계 밖 요구에 명확히 No

8.2 Next Steps

즉시 시작할 것

  1. 이해관계자 회의
    • 플랫폼팀 + SI팀 리드 + 경영진

    • 일정: 1주 내
